---
description: 전송 - 스테이징, 커밋, 푸시 한번에 진행
---

로컬 변경 사항을 스테이징, 커밋, 푸시까지 한번에 진행해주세요.

**작업 절차**:
1. git status로 변경 사항 확인
2. git diff로 변경 내용 검토
3. 변경 사항 분류:
   - 관련 내용별로 그룹화 (예: CSV 기능 관련 파일, 버그 수정 관련 파일)
   - 파일 내용, 경로, 변경 타입 분석
4. 각 그룹별로 스테이징:
   - 관련 파일들을 함께 add
   - 그룹별로 별도 커밋 생성
5. 커밋 메시지 자동 생성:
   - 각 그룹의 변경 내용 분석
   - [태그] 요약 형식으로 메시지 생성
   - 태그: [추가], [수정], [버그수정], [리팩토링], [문서], [테스트]
   - 요약은 한글로 간결하게 작성
6. 모든 커밋 완료 후 원격 저장소로 푸시
7. git status로 최종 상태 확인

**변경 사항 분류 규칙**:
1. **기능 단위**: 동일 기능과 관련된 파일은 한 그룹으로 처리
   - 예: ExportService.cs, ExportViewModel.cs, ExportTemplate.liquid
2. **버그 단위**: 동일 버그와 관련된 수정은 한 그룹으로 처리
   - 예: ExportService.cs 수정, ExportServiceTests.cs 수정
3. **문서 단위**: 문서 변경은 별도 그룹으로 처리
   - 예: README.md, BUILD_GUIDE.md
4. **테스트 단위**: 테스트 관련 파일은 한 그룹으로 처리
   - 예: ExportTests.cs, ExportServiceTests.cs

**태그 결정 규칙**:
- 새로운 파일이 포함된 경우: [추가]
- 버그 관련 수정인 경우: [버그수정]
- 기존 기능 개선/변경: [수정]
- 코드 구조 개선: [리팩토링]
- 문서만 변경: [문서]
- 테스트만 추가/수정: [테스트]

**요구사항**:
- 절대 규칙 위반 코드는 커밋하지 않음
- 불필요한 파일(.env, credentials.json 등)은 제외
- force push 금지 (main/master 브랜치)
- 충돌 시 사용자에게 알림

**예시**:

```bash
# 단일 그룹 변경
/전송
→ 분석 결과: ExportService.cs 추가, ExportViewModel.cs 수정 (CSV 기능)
→ 커밋 1: [추가] CSV 데이터 추출 기능
→ 푸시 완료

# 여러 그룹 변경
/전송
→ 분석 결과:
  1. ExportService.cs, ExportViewModel.cs (CSV 기능 추가)
  2. ExportServiceTests.cs (테스트 추가)
  3. README.md (문서 업데이트)
→ 커밋 1: [추가] CSV 데이터 추출 기능
→ 커밋 2: [테스트] CSV 추출 기능 테스트 추가
→ 커밋 3: [문서] CSV 기능 관련 문서 업데이트
→ 푸시 완료

# 버그 수정
/전송
→ 분석 결과: ExportService.cs 수정 (null 참조 수정)
→ 커밋 1: [버그수정] ExportService null 참조 예외 처리
→ 푸시 완료
```
