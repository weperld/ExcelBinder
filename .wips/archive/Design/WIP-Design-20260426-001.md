## 설계 결과

### 아키텍처 다이어그램

**데이터 흐름 (enum 컬럼 추가 후)**

```
[Schema JSON]                      [SchemaEditor UI]
"Element": "int:enum:Fire"  <─────  Data Type=int + Enum Type="Fire"
       │
       ▼
  TypeParser.ParseType()
       │
       ▼
  TypeInfo {
    BaseType="int",
    IsEnum=true,
    EnumType="Fire",
    ColumnName="Element"
  }
       │
       ├──────────────► ExportService.WriteField()
       │                   ├─ IsEnum? → writer.Write(엑셀셀값 문자열)
       │                   └─ else    → 기존 WritePrimitive
       │
       ├──────────────► ExportService.ParseField()
       │                   ├─ IsEnum? → return 엑셀셀값 그대로
       │                   └─ else    → 기존 ParsePrimitive
       │
       └──────────────► CodeGeneratorService.PrepareDataContext()
                           ├─ Type    = ConvertPrimitive(info)  → "Fire"
                           └─ ReadMethod = GetReadMethod(info)
                                          → "System.Enum.Parse<Fire>(reader.ReadString())"
```

### 기술적 결정 사항

#### 결정 1: TypeInfo 확장 방식 — **구조체 필드 추가 (현행 패턴 유지)**
- 대안: 별도 `EnumInfo` 클래스 추가 → 거부 (구조체 디자인 깨짐, ParseType 반환 일관성 깨짐)
- 채택: `IsEnum`/`EnumType` 필드 직접 추가. `default(TypeInfo)` 시 `IsEnum=false`, `EnumType=null`로 안전.

#### 결정 2: TypeParser의 `:enum:` 토큰 처리 순서 — **기존 split 후 마커 검사와 동일 패턴**
- 기존 `parts.Contains("ref")` → `parts.Contains("enum")` 동일 패턴 추가
- ColumnName fallback 조건(라인 43-47)에 `(EnumType == null || last != EnumType)` 추가하여 `MyEnum`이 ColumnName으로 잘못 잡히는 것 방지

#### 결정 3: ExportService 시그니처 유지 — **신규 private helper 추가**
- `WritePrimitive(string type, ...)` 시그니처 변경 시 caller `WriteField` 1곳만 영향이지만, 기존 동작 보존을 위해 helper(`WritePrimitiveOrEnum`)를 추가하고 `WriteField`만 호출 변경
- public 메서드(`ExportToBinary`, `ExportToJson`) 시그니처는 절대 변경 안 함 → StaticDataProcessor 호출부 무영향

#### 결정 4: CodeGeneratorService 시그니처 — **오버로드 추가**
- 기존 `private string GetReadMethod(string type)` 유지
- 신규 `private string GetReadMethod(TypeInfo info)` 오버로드 추가 → IsEnum이면 `Enum.Parse` 인라인, 아니면 기존 메서드에 위임
- `ConvertPrimitive(string, FeatureDefinition)`는 `ConvertPrimitive(TypeInfo, FeatureDefinition)`로 시그니처 변경 (호출부 2곳: PrepareDataContext, GetKeyType)

#### 결정 5: Read 코드 인라인 형태 — **System.Enum.Parse 풀네임 사용**
- 사용자 프로젝트의 `using System;` 의존성 회피
- 생성 코드: `System.Enum.Parse<MyEnum>(reader.ReadString())`
- List<enum>은 기존 List 루프 안에서 같은 표현식 사용 (템플릿이 ReadMethod 토큰을 재사용)

#### 결정 6: SchemaFieldViewModel.SelectedType setter — **setter 내부에서 EnumType 자동 초기화**
- 사용자가 Data Type을 `int` → `string`으로 바꾸면 EnumType 즉시 비워짐
- IsEnabled=False 만으로는 잔존 텍스트가 Save에서 토큰으로 인코딩될 위험 → 명시적 초기화로 방어

#### 결정 7: View 컬럼 너비 — **헤더와 데이터 행 동기화 필수**
- 두 Grid의 `ColumnDefinitions`가 동일해야 정렬 유지 → 양쪽 모두 (60/180/120/150/*) 적용

### 수렴 분석

**1회차 보완 사항 도출:**

| # | 보완 사항 | 분류 | 반영 여부 |
|---|----------|------|----------|
| 1 | `TypeInfo` 구조체에 `IsEnum`/`EnumType` 추가 시 기존 `default(TypeInfo)` 호출이 모두 안전한가? | 필수 | 검증: ParseType만 TypeInfo를 생성, 외부에서 default 인스턴스화 없음 → 안전 |
| 2 | TypeParser의 ColumnName fallback이 EnumType을 ColumnName으로 오인할 위험 | 필수 | 반영: fallback 조건에 `(EnumType == null || last != EnumType)` 추가 |
| 3 | EnumType이 비어있는데 IsEnum=true인 비정상 상태 차단 | 필수 | 반영: ParseType에서 `enum` 토큰 발견 시 다음 토큰 존재 여부 확인 후 EnumType 설정. 다음 토큰 없으면 IsEnum=false로 폴백 |
| 4 | CodeGeneratorService의 GetKeyType이 ConvertPrimitive(string)을 호출 → TypeInfo 시그니처로 변경 시 호출부 갱신 필요 | 필수 | 반영: GetKeyType에서 `ParseType` 결과를 `info` 그대로 ConvertPrimitive(info,…)에 전달 |
| 5 | List<int:enum:MyEnum> 처리 시 LIST 안의 BaseType과 EnumType이 모두 보존되는지 | 필수 | 검증: List 토큰 제거(Substring) 후 내부 문자열에 split 적용 → enum 토큰 정상 파싱. 추가 코드 불필요 |
| 6 | Reference 입력란과 Enum 입력란이 동시에 활성될 수 있음 (`int:enum:E:ref:R`) | 선택 | 사용자 동시 입력 허용. UX 우려는 있으나 제약 추가 시 복잡도 ↑ → 추후 검토 |
| 7 | EnumType 입력 후 Data Type을 string 등 비정수형으로 변경 시 EnumType 자동 초기화 안 되면 Save에서 잘못된 토큰 생성 | 필수 | 반영: SelectedType setter에서 `if (!CanInputEnum) EnumType = string.Empty` |
