#### 5. `ViewModels/MainViewModel.cs` (수정)
```csharp
private FeatureGroupService _groupGroupService;
private FeatureGroup? _selectedGroup;

public ObservableCollection<FeatureGroup> Groups { get; } = new();

public FeatureGroup? SelectedGroup
{
    get => _selectedGroup;
    set
    {
        if (SetProperty(ref _selectedGroup, value))
        {
            OnPropertyChanged(nameof(FilteredFeatures));
        }
    }
}

public IEnumerable<FeatureDefinition> FilteredFeatures
{
    get
    {
        if (_selectedGroup == null || _selectedGroup.IsAllGroup)
            return Features;
        var ids = new HashSet<string>(_selectedGroup.FeatureIds);
        return Features.Where(f => ids.Contains(f.Id));
    }
}

// 신규 커맨드
public ICommand AddGroupCommand { get; }                          // 사이드바 빈공간/+ 버튼 양쪽
public ICommand RenameGroupCommand { get; }                       // 파라미터: FeatureGroup
public ICommand DeleteGroupCommand { get; }                       // 파라미터: FeatureGroup
public ICommand ToggleFeatureGroupMembershipCommand { get; }      // 파라미터: (FeatureGroup, FeatureDefinition) 튜플 또는 wrapper

private void RefreshGroups()
{
    Groups.Clear();
    Groups.Add(new FeatureGroup {
        Id = ProjectConstants.Defaults.AllGroupId,
        Name = ProjectConstants.Defaults.AllGroupName
    });
    var loaded = _groupService.LoadGroups();
    foreach (var g in loaded) Groups.Add(g);
    SelectedGroup = Groups.FirstOrDefault();
}
```

### 기술적 결정 사항

#### 결정 1: "전체" 그룹의 가상성 — **런타임 컬렉션 첫 번째 항목으로 강제 삽입**
- `_groups.json`에는 저장 X
- `RefreshGroups()` 시작 시 항상 첫 번째로 `Add(전체 그룹)` → 사용자 그룹 로드
- `IsAllGroup` 속성 (계산속성, `Id == "__all__"`) 으로 식별

#### 결정 2: MainViewModel.CustomGroups 추가 (서브메뉴용 필터링된 컬렉션)
- Feature ContextMenu 서브메뉴는 "전체" 그룹을 제외해야 함
- `Groups`에서 IsAllGroup 제외한 별도 ICollectionView 또는 ObservableCollection 노출
- 또는 XAML CollectionViewSource Filter 사용
- → **권장: `CustomGroups`라는 별도 ObservableCollection 노출** (단순)

#### 결정 3: ToggleFeatureGroupMembershipCommand 파라미터 전달
- `(group, feature)` 튜플이 필요
- WPF MultiBinding을 활용한 ValueTuple 전달은 복잡 → **MenuItem.Tag로 Feature 전달, CommandParameter로 Group 전달, 또는 별도 wrapper 클래스**
- → **권장 패턴**: `MenuItemFeatureGroupBinding` wrapper (group + feature 보유) 또는 ContextMenu Opened 이벤트 핸들러로 동적 빌드
- 단순화 방안: MenuItem.IsChecked 바인딩은 IsCheckedConverter (group.FeatureIds.Contains(feature.Id)), Click 이벤트 핸들러 (코드비하인드 X 원칙이지만 WPF ContextMenu의 한계로 허용)

→ **최종 채택**: **코드비하인드 없는 패턴 우선** — 각 그룹별 MenuItem을 코드(MainViewModel)에서 동적 빌드하는 대신, `MenuItemViewModel { Name, IsChecked, ToggleCommand }` 같은 **wrapper VM 컬렉션을 Feature ContextMenu Opened 시점에 생성**

→ **단순화 최종 채택**: ContextMenu Opened 이벤트에서 코드 비하인드 핸들러로 MenuItem 동적 생성. **이 한 군데만 코드비하인드 허용** (WPF 한계). 핸들러는 MainViewModel의 Groups를 순회하여 Header/IsChecked/Click을 설정.

#### 결정 4: GroupContextMenu 전용 코드비하인드 — **DashboardView.xaml.cs에 ContextMenu_Opened 핸들러 1개만 허용**
- 정당화: WPF ContextMenu의 동적 ItemsSource + IsChecked + Command 조합은 매우 복잡, MVVM 순수성을 위해 wrapper VM을 늘리느니 1개 코드비하인드가 깔끔
- 핸들러는 그룹 수가 많아도 O(N) 단순 루프
- 각 MenuItem은 `IsChecked` 직접 설정, `Click` 시 ToggleFeatureGroupMembershipCommand 실행

#### 결정 5: 그룹 추가/이름변경 UI — **DialogService.ShowInput 사용**
- ShowInput(title, prompt, defaultValue, validator) 시그니처
- validator는 그룹명 중복/빈문자열 검증 로직 주입
- ShowUpdateInfoDialog와 동일한 코드 기반 Window 패턴

#### 결정 6: 데이터 저장 시점 — **모든 그룹 변경은 즉시 SaveGroups() 호출**
- 그룹 추가/이름변경/삭제/Feature 토글 시점마다 즉시 파일 저장
- 단순, 데이터 유실 위험 ↓

| 메모리 누수 없음 | ✅ | IDisposable 자원 없음, 이벤트 구독은 Settings 변경 시만 |
| 데이터 무결성 | ✅ | 토글 즉시 저장, lazy GC로 dangling ID 무시 |
| UI 프리징 방지 | ✅ | 그룹 파일 I/O는 ms 단위, Task.Run 불필요 |
| 아키텍처 준수 | ✅ | MVVM 유지, ContextMenu 코드비하인드는 WPF 한계 영역으로 한정 (1개 핸들러만) |
